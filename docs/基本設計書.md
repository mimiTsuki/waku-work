## 1. システム概要

### 1.1 目的

案件別の稼働時間をローカル環境で記録・集計し、ジョブカンへの週次転記作業を補助する。データはすべてローカルのJSONファイルで管理し、外部サービスへの送信は一切行わない。

### 1.2 技術スタック

| 役割                 | 採用技術                        | バージョン（目安） |
| -------------------- | ------------------------------- | ------------------ |
| アプリケーション基盤 | Electron + electron-vite        | Electron 30系      |
| レンダラープロセス   | React                           | 18系               |
| UIコンポーネント     | shadcn/ui                       | 最新安定版         |
| 日時計算             | date-fns                        | 3系                |
| 状態管理             | useState / useReducer + Context | —                  |
| ファイルI/O          | Node.js fs（Electron組み込み）  | —                  |

### 1.3 動作環境

- macOS / Windows（Electronがサポートするデスクトップ環境）
- データ保存先：ユーザーが設定画面で任意指定するローカルディレクトリ
  - 仕様書はmacOS準拠で記載(ファイルパスなど)

### 2.2 セキュリティ方針

- `contextIsolation: true`、`nodeIntegration: false` を維持し、Webセキュリティモデルを遵守する。
- レンダラープロセスからのファイルアクセスは、`contextBridge` で公開した `window.electronAPI` 経由のIPC呼び出しに限定する。
- レンダラープロセスはNode.js APIに直接アクセスしない。

### 3.3 SCR-001 ログ入力画面（週間カレンダービュー）

Google カレンダー・Toggl Track を参考にした、時間軸 × 曜日のタイムライングリッドをメインUIとする。

#### 3.3.1 全体レイアウト

```
┌─────────────────────────────────────────────────────────────────────┐
│ [< 前週]  2026年2月16日（月）〜 2月22日（日）  [次週 >]  [今週]       │
├────────┬──────────┬──────────┬──────────┬──────────┬─────┬──────────┤
│        │  月 2/16 │  火 2/17 │  水 2/18 │  木 2/19 │ ... │  日 2/22 │
├────────┼──────────┼──────────┼──────────┼──────────┼─────┼──────────┤
│  9:00  │          │▓▓▓▓▓▓▓▓▓│          │          │     │          │
│        │          │〇〇案件  │          │          │     │          │
│ 10:00  │▓▓▓▓▓▓▓▓▓│▓▓▓▓▓▓▓▓▓│          │          │     │          │
│        │△△案件  │〇〇案件  │          │          │     │          │
│ 11:00  │▓▓▓▓▓▓▓▓▓│          │          │          │     │          │
│        │△△案件  │          │          │          │     │          │
│ 12:00  │          │          │          │          │     │          │
│  ...   │          │          │          │          │     │          │
└────────┴──────────┴──────────┴──────────┴──────────┴─────┴──────────┘
```

#### 3.3.2 タイムライングリッドの仕様

| 項目           | 仕様                                                                      |
| -------------- | ------------------------------------------------------------------------- |
| 表示時間帯     | デフォルト表示は 09:00〜20:00。スクロールで 00:00〜23:00 全体にアクセス可 |
| 時間軸の粒度   | 1時間ごとにラベル表示、30分ごとに細い区切り線を表示                       |
| 週の定義       | 月曜日〜日曜日                                                            |
| ナビゲーション | 「前週」「次週」ボタン、および「今週」クイックボタン                      |
| 列幅           | 7列均等分割（曜日ヘッダーに「曜 M/D」の形式で日付を併記）                 |

#### 3.3.3 稼働ブロックの表示仕様

- ログは案件の設定カラーで塗られた**ブロック**としてグリッド上に描画する。
- ブロック内には「案件名」「開始〜終了時刻」を表示する。ブロックの高さが不足して文字が入らない場合は案件名のみ表示する。
- 同一日・同一時間帯に複数ブロックが重なる場合、列を分割して横並びで表示する（Google カレンダーと同方式）。

```
例）2/17 の 09:00〜11:00 に2件重複する場合

│  火 2/17              │
├───────────┬───────────┤
│ 〇〇案件  │ △△案件  │
│ 09:00     │ 09:30     │
│ 〜11:00   │ 〜10:30   │
│           │           │
└───────────┴───────────┘
```

#### 3.3.4 インタラクション

| 操作                                             | 動作                                                                                                                         |
| ------------------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------- |
| グリッド上の空白セルをクリック                   | 新規ログ追加モーダルを開く。クリックした日付・時刻を初期値にセット                                                           |
| グリッド上の空白セルをドラッグ（上→下）          | ドラッグした範囲をプレビューブロックとして即時表示。マウスアップ時に新規追加モーダルを開く（開始・終了時刻を初期値にセット） |
| 既存ブロックをクリック                           | 編集モーダルを開く                                                                                                           |
| 既存ブロックの本体をドラッグ                     | ブロックを移動。別の曜日列へのドロップで日付変更も可能。ドロップ時に即時保存                                                 |
| 既存ブロック下端のリサイズハンドルをドラッグ     | 終了時刻を変更。ドロップ時に即時保存                                                                                         |
| 既存ブロックにホバー → 表示される「×」をクリック | 削除確認ダイアログを表示                                                                                                     |

**ドラッグ中の共通仕様**

- 時刻のスナップ粒度は **15分単位** とする（例: 10:13 → 10:15、10:07 → 10:00）。
- ドラッグ中はブロックをゴースト表示（半透明）し、ドロップ先の時間帯をツールチップで表示する。
- `endTime ≤ startTime` になる操作（ブロックの本体移動・リサイズ共通）はスナップを無効化してドラッグを制限し、最低 15分の高さを保証する。
- 深夜またぎ（24:00を超える）になる操作はグリッド下端でドラッグを停止し、23:45〜23:59 を最大値とする。
- ドラッグ中は `pointer-events: none` を列全体に付与してテキスト選択を抑制する。

#### 3.3.5 ログ追加 / 編集モーダル

```
┌──────────────────────────────────────────┐
│  稼働ログを追加                            │
├──────────────────────────────────────────┤
│  日付      [ 2026-02-17            ▼ ]   │
│                                          │
│  案件      [ 〇〇案件              ▼ ]   │
│                                          │
│  開始時刻  [ 10 ]:[ 00 ]                 │
│  終了時刻  [ 11 ]:[ 00 ]                 │
│                      計: 1:00            │
│                                          │
│  メモ      [                           ] │
│            [                           ] │
│                                          │
│              [キャンセル]     [保存]      │
└──────────────────────────────────────────┘
```

- **日付**: shadcn/ui `Popover` + `Calendar` コンポーネント
- **案件**: shadcn/ui `Select`（アーカイブ済みは表示しない）

- 期間が複数月にまたがる場合、対象月数分のIPCリクエストを `Promise.all` で並列発行してマージする。

## 4. 機能設計

### 4.1 稼働ログCRUD

#### バリデーション仕様

| 項目     | 必須 | バリデーション内容                                           |
| -------- | ---- | ------------------------------------------------------------ |
| 日付     | ○    | 有効な日付であること                                         |
| 案件     | ○    | アクティブな案件が1件選択されていること                      |
| 開始時刻 | ○    | `HH:mm` 形式、00:00〜23:59 の範囲                            |
| 終了時刻 | ○    | `HH:mm` 形式、かつ開始時刻より後であること（深夜またぎ不可） |
| メモ     | —    | 最大500文字（任意）                                          |

#### ログIDの採番

`log_` + `Date.now()` で生成する（例: `log_1708329600000`）。ユーザー単独の手動入力ツールという特性上、同一ミリ秒衝突は実用上問題なしと判断する。

#### 保存処理フロー

```
レンダラー
  1. フォームのバリデーション実行
  2. window.electronAPI.writeLogs(year, month, logs) を呼び出す

メインプロセス
  3. logs/YYYY-MM.json を読み込む（存在しない場合は [] として扱う）
  4. 新規追加: エントリーを末尾に追加
     編集:    該当IDのエントリーを差し替え
     削除:    該当IDのエントリーを除外
  5. エントリーを date → startTime の昇順でソート
  6. インデント2スペースで整形してファイルを上書き保存

レンダラー
  7. 成功レスポンスを受けてUIの状態を更新
```

### 4.2 週間カレンダービューの描画

#### ブロック位置計算

```
ブロックの top    = (startTime - 基準時刻) × (1時間あたりのpx高さ / 60)
ブロックの height = (endTime - startTime) × (1時間あたりのpx高さ / 60)
```

- 1時間あたりの高さは `60px` を基準値とし、CSS変数で定義する。
- 基準時刻は `00:00` とし、グリッド全体を0〜23時の範囲で描画したうえで、デフォルト表示時に07:00相当のスクロール位置に初期スクロールする。

#### 重複ブロックの列分割アルゴリズム

1. 同一日のブロックを開始時刻の昇順でソートする。
2. 時間帯が重なるブロック群（オーバーラップグループ）を検出する。
3. グループ内の各ブロックに列インデックス（0, 1, 2...）を割り当てる（貪欲法）。
4. `width = 列の幅 ÷ グループ内のブロック数`、`left = 列インデックス × width` でCSSを設定する。

### 4.3 ドラッグ操作

ドラッグ操作はすべてマウスイベント（`onMouseDown` / `onMouseMove` / `onMouseUp`）で実装し、HTML5 Drag and Drop APIは使用しない（グリッド座標への精密な位置計算が困難なため）。

#### 4.3.1 ドラッグによるブロック生成

```
1. [mousedown] 空白セル上でドラッグ開始
   → ドラッグ開始位置からY座標を startTime として記録

2. [mousemove] ドラッグ中
   → 現在のY座標を15分単位にスナップし endTime として算出
   → startTime〜endTime のプレビューブロック（半透明）をDayColumn上にオーバーレイ表示
   → ツールチップで「HH:mm 〜 HH:mm」を表示

3. [mouseup] ドラッグ終了
   → プレビューブロックを消去
   → startTime == endTime（クリック判定、5px未満のドラッグ）の場合はモーダルを開かずクリック処理に移譲
   → startTime < endTime の場合、LogFormModal を開く（日付・開始時刻・終了時刻を初期値にセット）
```

#### 4.3.2 ドラッグによるブロック移動

```
1. [mousedown on LogBlock 本体] ドラッグ開始
   → ブロックをゴースト化（opacity: 0.5）
   → ドラッグ開始時のカーソルとブロック上端のオフセット（offsetY）を記録

2. [mousemove] ドラッグ中
   → カーソルのX座標から対象の曜日列（date）を算出
   → カーソルのY座標 - offsetY から startTime を15分単位にスナップして算出
   → endTime = startTime + 元の稼働時間（分） で維持
   → ドロップ先のゴーストブロックを描画し、ツールチップで「M/D HH:mm 〜 HH:mm」を表示

3. [mouseup] ドラッグ終了
   → 変更後の date / startTime / endTime でエントリーを更新
   → 月をまたいだ場合は旧月ファイルから削除・新月ファイルへ追加の2回書き込みを行う
   → ファイル保存完了後にUIの状態を更新
```

#### 4.3.3 ドラッグによるリサイズ（終了時刻変更）

```
1. [mousedown on リサイズハンドル] ドラッグ開始
   → ブロック下端に高さ8pxのハンドルを常時表示（ホバーで視覚強調）
   → cursor: ns-resize を付与

2. [mousemove] ドラッグ中
   → カーソルのY座標から endTime を15分単位にスナップして算出
   → endTime > startTime + 15分 の制約を維持（最小ブロック高さを保証）
   → ブロックのheightをリアルタイム更新し、ツールチップで「〜 HH:mm（X:XX）」を表示

3. [mouseup] ドラッグ終了
   → 変更後の endTime でエントリーを更新・保存
```

#### 4.3.4 共通の実装方針

- 3種のドラッグ操作はそれぞれ `useDragCreate`、`useDragMove`、`useDragResize` カスタムフックに分離し、`DayColumn` / `LogBlock` から呼び出す。
- `mousemove` / `mouseup` リスナーはドラッグ開始時に `document` に登録し、終了時に解除する（グリッド外へのドラッグアウトに対応するため）。
- ドラッグ中の状態は `useDragState` で一元管理し、Context経由で `WeekCalendar` 全体に共有する。

```typescript
// ドラッグ状態の型
type DragState =
  | { type: 'idle' }
  | { type: 'creating'; date: string; startTime: string; currentEndTime: string }
  | { type: 'moving'; entryId: string; date: string; startTime: string; endTime: string }
  | { type: 'resizing'; entryId: string; endTime: string }
```

#### 4.3.5 スナップ計算

```typescript
// Y座標 → 時刻（15分スナップ）
function snapToTime(y: number, hourHeight: number): string {
  const totalMinutes = Math.round(((y / hourHeight) * 60) / 15) * 15
  const h = Math.floor(totalMinutes / 60)
  const m = totalMinutes % 60
  return `${String(h).padStart(2, '0')}:${String(m).padStart(2, '0')}`
}
```

### 4.4 集計ロジック

```
1. 指定期間内のログを date-fns でフィルタリング
2. [date, projectId] のペアをキーにグルーピング
3. 同グループ内の各エントリーの (endTime - startTime) を分単位で合算
4. 合計分を H:mm 形式に変換して表示
```

集計処理はレンダラープロセス側で行う。期間が複数月にまたがる場合は対象月数分のIPCリクエストを `Promise.all` で並列発行し、結果をフラットにマージしてから集計する。

### 4.5 案件マスター管理

- **削除機能は提供しない**。案件の無効化はアーカイブ（`archived: true`）のみとする。
- アーカイブされた案件は集計に引き続き反映され、過去の稼働データの整合性を保つ。

### 4.6 設定管理

- アプリ起動時にメインプロセスが `~/.config/waku-work/settings.json` を読み込む。
- 未設定の場合は `~/.config/waku-work/logs` をデフォルトとし、ディレクトリが存在しなければ自動で作成する。
- データ保存ディレクトリ変更時の既存ファイルの自動移行は提供しない（ユーザーが手動でコピーする）。

## 6. データ設計

### 6.1 型定義

```typescript
// 案件マスター
interface Project {
  id: string // "proj_" + タイムスタンプ
  name: string // 案件名（最大50文字）
  color: string // カラーコード（プリセット12色のいずれか）
  archived: boolean // アーカイブフラグ
}

// 稼働ログエントリー
interface LogEntry {
  id: string // "log_" + タイムスタンプ
  date: string // "YYYY-MM-DD"
  projectId: string // Project.id への参照
  startTime: string // "HH:mm"（endTime > startTime を保証、深夜またぎなし）
  endTime: string // "HH:mm"
  memo: string // 任意メモ（最大500文字、未入力時は ""）
  createdAt: string // ISO 8601形式
}

// アプリ設定
interface AppConfig {
  dataDir: string // データ保存ディレクトリの絶対パス
}
```

### 6.2 ファイル構成

```
{dataDir}/
├── projects.json         # 案件マスター（全件）
└── logs/
    ├── 2026-01.json      # 2026年1月の稼働ログ
    ├── 2026-02.json      # 2026年2月の稼働ログ
    └── ...
```

### 6.3 ファイルI/O方針

- 読み書きは「ファイル全体を読み込み → メモリ上で加工 → 全体を上書き保存」の方式とする。
- 保存するJSONはインデント2スペースで整形し、git diffの可読性を確保する。
- ログファイルは `date` → `startTime` の昇順でソートして保存する。
- ファイルが存在しない場合（新規月など）は `[]` として扱い、書き込み時にファイルを新規作成する。
- 書き込み前のバックアップは行わない（gitによるバージョン管理を前提とする）。

## 8. エラーハンドリング方針

| エラー発生箇所         | エラー種別                       | ハンドリング方法                                                                    |
| ---------------------- | -------------------------------- | ----------------------------------------------------------------------------------- |
| ファイル読み込み失敗   | ディレクトリ未存在、権限エラー等 | メインプロセスでcatchし、エラーメッセージをレンダラーに返す。UIにトースト通知で表示 |
| ファイル書き込み失敗   | ディスク容量不足、権限エラー等   | 同上                                                                                |
| JSONパースエラー       | ファイル破損等                   | 空配列として扱わず、エラーとしてUIに通知する（データ破損の隠蔽を防ぐ）              |
| バリデーションエラー   | 不正入力                         | フォーム内にインラインメッセージとして即時表示                                      |
| 設定ディレクトリ未設定 | 初回起動時                       | デフォルトパスを自動設定して起動する                                                |
