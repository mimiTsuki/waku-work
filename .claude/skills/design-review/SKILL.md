# Design Review

コードベースを探索し、設計上の改善点を分析・提案するスキル。

## 対象

- `$ARGUMENTS` が指定されている場合: そのパスを対象にする
- 指定がない場合: `src/` 全体を対象にする

## 探索手順

### Step 1: ファイル構成の把握

1. 対象ディレクトリのファイル一覧を取得する（Glob / LS）
2. 主要モジュールと各ファイルの責務を理解する

### Step 2: 実装の詳細確認

1. 重要なファイルを読み込み、実装の詳細を確認する
2. モジュール間の依存関係を把握する

### Step 3: 分析

以下の観点で分析する:

- **アーキテクチャ**: 責務の分離、依存方向、レイヤー違反、不要な間接層
- **コード品質**: 重複、命名、一貫性、エラーハンドリング
- **型安全性**: any の使用、型の網羅性、不十分な型定義

### Step 4: Issue ファイルの作成

1. `docs/issues/` 内の既存ファイルを確認し、最大の番号を特定する
2. 重複する既存 Issue がないか確認する（同じ問題を二重に作成しない）
3. 新しい Issue ごとに連番でファイルを作成する

## 出力形式

各 Issue は `docs/issues/NNN-slug.md` に以下の形式で作成する:

````md
# [問題のタイトル]

**Priority**: High / Medium / Low
**File**: `ファイルパス`

## Problem

何が問題かを説明する。

該当コードの引用:

\```ts
// 問題のあるコード
\```

## Suggestion

どう改善すべきかを説明する。

改善後のコード例（あれば）:

\```ts
// 改善案
\```

## Consideration

- トレードオフや注意点があれば記載する
````

## 注意事項

- AGENTS.md のアーキテクチャルール（依存方向など）を判断基準に含めること
- 些細なスタイルの問題は報告しない（リンターで検出できるものは除外）
- 既存の docs/issues/ と重複する指摘は作成しない
- 各 Issue は独立して読めるように十分なコンテキストを含めること
- 最後に、作成した Issue の一覧をサマリとしてユーザーに表示すること
